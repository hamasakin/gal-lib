# Quick 20260512c — MetadataPicker 遮罩 / 年份兜底 / 复核队列回灌

## 背景

真机使用反馈三件事：
1. 截图：MetadataPicker 打开时背景 ReviewQueue 内容透过 overlay 渗进对话框两侧，视觉混乱
2. 大部分游戏 release_year 仍为空
3. Scan 页 ReviewQueue 看不到老库历史 `metadata_source='none'` 的游戏，用户无从手动绑

## Bug A — MetadataPicker DialogOverlay 太透明

### 根因

`src/components/ui/dialog.tsx:42` 的 `DialogOverlay` 用 `bg-black/10` —— 10% 不透明黑色，几乎全透明。背景 Scan 页 ReviewQueue 的 ExpandedCompare（双源候选 + 取消/应用按钮）从对话框外两侧渗出来，和 MetadataPicker 内部候选项视觉混叠。

### 修复

`dialog.tsx:42` overlay className 从 `bg-black/10 ... supports-backdrop-filter:backdrop-blur-xs` 改成 `bg-black/60 ... supports-backdrop-filter:backdrop-blur-sm`。提高遮罩不透明度 + 加 blur 强度。

风险：项目其它 Dialog 使用方都会受影响。审查所有 `<Dialog>` 使用点——目前只有 MetadataPicker 一处明显模态，其余 shadcn 派生（confirm/prompt 等）的视觉一致性增强是好事。

## Bug B — 年份依然空：Bangumi infobox 发售日 fallback

### 根因

Quick 20260512b 已经把 `release_year` 五处 UPDATE 路径都修了，但 Bangumi 顶层 `date` 字段在很多 game subjects 是空字符串（如 doujin / 老游戏 / 未发售）。`parse_release_year("")` 返回 None → `COALESCE(NULL, release_year)` 保持原值 NULL。

实际上 Bangumi infobox 经常带 `发售日: 2020-04-24` / `発売日: 2020年4月24日` / `发行日期: ...` 字段。

### 修复

在 `src-tauri/src/metadata/bangumi.rs:fetch_detail` 里加一个 `extract_release_date_from_infobox` helper：
- 扫描 keys：`发售日` / `发行日` / `発売日` / `发行日期` / `发售日期`
- value 用 `infobox_value_to_string`（已存在）取出字符串
- 容忍三种格式：`YYYY-MM-DD` / `YYYY年M月D日` / `YYYY-MM` / `YYYY`
- 返回 `Option<String>`，调用方在顶层 `date` 为空/None 时 fallback

修改 `MetadataDetail.release_date` 的赋值：`raw.date.filter(|s| !s.is_empty()).or_else(|| extract_release_date_from_infobox(...))`。

`parse_release_year` 已存在，无需改。

## Bug C — 复核队列回灌（reseed）

### 根因

`sync_review_queue_for_game` 只在新 ingest 事件（scan / add_game / bind / refresh）触发，把 needs_review (metadata_source='none' OR confidence<80) 的游戏 INSERT OR REPLACE 进 `scan_review_queue` 表。

**老库历史**：在 Phase 12 之前 ingest 进库的 unmatched 游戏 **从未经过** sync_review_queue_for_game，所以它们不在队列表里。用户在 Scan 页只看到新扫描的低置信度项，无从对老库的 unmatched 项手动绑定。

### 修复

新增 IPC `reseed_review_queue() -> i64`（返回灌入数量）：

```rust
// 把所有 needs_review 的 games 一次性 INSERT OR REPLACE 进 queue
INSERT OR REPLACE INTO scan_review_queue (game_id, game_path, current_confidence, suggested_source, suggested_id, created_at)
SELECT id, path, COALESCE(match_confidence, 0),
       CASE WHEN metadata_source IN ('bangumi','vndb') THEN metadata_source ELSE NULL END,
       CASE metadata_source WHEN 'bangumi' THEN bangumi_id WHEN 'vndb' THEN vndb_id ELSE NULL END,
       strftime('%Y-%m-%dT%H:%M:%fZ','now')
FROM games
WHERE metadata_source = 'none'
   OR (metadata_source != 'manual' AND COALESCE(match_confidence, 0) < 80);
```

`INSERT OR REPLACE` 保证已在队列的项目（已 dismiss 用户重新触发回灌想看见）也会被刷新 created_at，UI 排序自动靠前。

**注意**：`metadata_source != 'manual'` 排除手动绑定（manual = 用户已确认，永远不该回到 review）。

### 前端入口

`src/routes/Scan.tsx`：
- 顶部 actions 后面新增按钮 `重新生成待复核队列`（图标 ListRestart / RefreshCcw）
- 点击 → 调用 `reseed_review_queue` → toast 显示灌入数 → ReviewQueue 自动刷新（它已订阅 `meta-fetch-progress` / `scan-progress` 事件，但 reseed 不触发这些事件；额外手动触发一次 ReviewQueue 的 refetch —— 通过 `onMutated` 回调或事件 emit `review-queue-reseeded`）

简化方案：emit 一个新事件 `review-queue-reseeded`，让 ReviewQueue 监听并 refetch。或者更简单：用 `onMutated` 模式——但 ReviewQueue 是 Scan 的子组件，Scan 需要拿到 ReviewQueue 的 refetch handle。

最简：在 Scan.tsx 加一个 `reseedSeq` state，传给 ReviewQueue 作为 prop `reseedSeq`，ReviewQueue 在 effect 依赖 reseedSeq 变化时 refetch。
