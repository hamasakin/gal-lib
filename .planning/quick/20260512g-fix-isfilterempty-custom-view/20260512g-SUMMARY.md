# Quick 20260512g — Summary

## 变更

- `src/routes/Library.tsx:83-95` — `isFilterEmpty` 增加 `custom_view_id == null`
  与 `age_ratings == null || age_ratings.length === 0` 两条判定。

## 根因复盘

Library.tsx 的 `isFilterEmpty` 用来决定"这个 filter 要不要传给后端"。它的字段
列表比 `SearchFilter` 接口少了两个：

| SearchFilter 字段 | isFilterEmpty 列入 |
|---|---|
| tag_id | ✓ |
| status | ✓ |
| favorite | ✓ |
| brand | ✓ |
| year_decade | ✓ |
| brands | ✓ |
| staff_ids | ✓ |
| official_tags | ✓ |
| **custom_view_id** | ✗ ← 本次修 |
| **age_ratings** | ✗ ← 顺手 |

当 Sidebar 调 `applyFilter({ custom_view_id: id })`，refetchGrid 算出来的
`filterArg = isFilterEmpty(merged) ? null : merged`，由于 `isFilterEmpty` 错判
为空，结果是 `searchGames(query, sort, null)` —— filter 整个被丢，后端
按全库返回。后端 (`commands.rs:2043`) 实际支持 `custom_view_id`，所以根因 100%
在前端。

## 经验

`SearchFilter` 接口扩字段时，必须同步更新 `isFilterEmpty`。两个字段
（`custom_view_id` 来自 quick-20260510b，`age_ratings` 来自 quick-20260510b）
当时漏接，直到现在才被发现 —— 因为它们都是新近加入的过滤维度，平时点
"图书馆全景"或状态筛选不会触发。

后续考虑：把 `isFilterEmpty` 改成基于 `Object.entries` 的通用判空，或者干脆
让后端无条件接收 filter 对象（小代价节省一类隐性 bug）。本次不做，保持
最小修复。
