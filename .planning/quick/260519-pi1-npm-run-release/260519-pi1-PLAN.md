新建 `scripts/release.mjs`（Node ESM，纯标准库 `node:fs` / `node:child_process` / `node:os` / `node:path` / `node:process`，不引入任何新 npm 依赖）。脚本顶部加一段简短注释说明用法（`npm run release` / `npm run release -- 0.3.0` / `-- minor` / `-- major`）。失败时一律 `process.exit(1)` 并打印清晰中文提示。

按以下规格逐条实现：

**1. 参数解析**
- 通过 `process.argv.slice(2)` 取参数（`npm run release -- X` 时 X 会传到这里）。
- 读「当前版本」：解析 `package.json` 的 `version` 字段（用正则提取 `"version"` 那一行的值，不要 require）。当前版本必须是合法 `X.Y.Z`。
- 无参 → patch bump（`0.2.3` → `0.2.4`）。
- 参数为 `patch` / `minor` / `major` 关键字 → 对应语义 bump（minor: `0.2.3`→`0.3.0`，归零 patch；major: `0.2.3`→`1.0.0`，归零 minor+patch）。
- 参数为完整版本号（匹配 `^\d+\.\d+\.\d+$`）→ 直接采用为目标版本。
- 其他任何参数、或解析出的目标版本非法 → 打印中文错误（说明合法用法）并 `exit(1)`。
- 目标版本不得与当前版本相同。

**2. 前置检查**（任一不满足 → 打印对应中文错误并 `exit(1)`，此时尚未做任何文件改动）
- ① 工作区干净：`git status --porcelain` 输出为空。
- ② 当前分支为 `master`：`git rev-parse --abbrev-ref HEAD` 等于 `master`。
- ③ 目标 tag `vX.Y.Z` 本地与远端均不存在：
  - 本地 — `git tag --list vX.Y.Z` 输出为空；
  - 远端 — `git ls-remote --tags origin refs/tags/vX.Y.Z` 输出为空。
- 三项检查必须全部在做任何文件写入之前完成。

**3. Bump 四处版本字段**（旧版 `OLD` → 新版 `NEW`，全部用「精确字符串/正则替换」保持 diff 最小，每个文件只动 version 那一行；不要用 JSON.parse/stringify 以免重排格式）
- `package.json` — 替换 `"version": "OLD"` → `"version": "NEW"`（顶层 version 字段那一行）。
- `src-tauri/tauri.conf.json` — 同上替换 `"version": "OLD"` → `"version": "NEW"`。
- `src-tauri/Cargo.toml` — 替换 `[package]` 段内的 `version = "OLD"` → `version = "NEW"`。用锚定正则确保只命中 `[package]` 段那一行（例如匹配文件开头 `[package]` 之后、下一个 `[` 段之前的 `version = "..."`），不要误伤其它段。
- `src-tauri/Cargo.lock` — **只改 gal-lib 自身这一个 `[[package]]` 条目**。用正则锚定 `name = "gal-lib"` 紧随的下一行 `version = "..."` 做替换（例如 `/(name = "gal-lib"\r?\nversion = ")[^"]+(")/`）。Cargo.lock 中其余所有依赖条目（`name = "<dep>"` + `version`）一律不动。这是 v0.2.1/v0.2.2/v0.2.3 三次发版反复确认的硬约束。
- 每处替换前校验旧值确实命中（替换计数为 1）；命中数不为 1 → 报错中止（说明哪个文件没匹配上，可能版本字段格式已变），此时若已写过部分文件需提示用户用 `git checkout -- <files>` 回滚。
- 读写文件保留原换行风格（不要强制转 LF，Windows 下 Cargo.lock 等可能是 CRLF）。

**4. 提交**
- `git add package.json src-tauri/tauri.conf.json src-tauri/Cargo.toml src-tauri/Cargo.lock`
- `git commit -m "chore: bump version to NEW"`
- commit 成功后记录「commit 已完成」状态，供后续失败处理引用。

**5. 生成 tag 消息草稿 + 编辑器定稿**
- 求上一个 tag：`git describe --tags --abbrev=0 --match "v*" HEAD`（取 HEAD 可达的最近 v* tag）。若仓库无任何 tag（命令失败）→ 草稿提交列表退化为「全部历史」或留空，不要因此中止。
- 草稿临时文件写入 `os.tmpdir()`（文件名形如 `gal-lib-release-vNEW.txt`）。内容：
  - 第 1 行：`Release vNEW`
  - 第 2 行：空行
  - 之后：`git log <上个tag>..HEAD --pretty=format:"- %s"` 的输出（上个 tag 不存在时用 `git log --pretty=format:"- %s"`）。
- 打开编辑器让用户定稿，编辑器优先级：`git config core.editor` → `process.env.GIT_EDITOR` → `process.env.EDITOR` → `process.env.VISUAL` → Windows 兜底 `notepad`。
- 用 `child_process.spawnSync` 同步等待编辑器进程退出（`stdio: 'inherit'`），把临时文件路径作为参数传给编辑器。Windows 下 notepad 直接可作为命令；core.editor 可能含参数，需按空格拆分命令与参数。
- 编辑器非 0 退出 → 报错中止（提示 commit 已完成，可手动 `git tag -a vNEW` 后续收尾）。

**6. 打 tag**
- 编辑器退出后读回临时文件内容；去掉以 `#` 开头的注释行与首尾空白后，若内容为空 → 报错「tag 消息为空，已取消打 tag」并 `exit(1)`（提示 commit 已完成的收尾方式）。不发空 notes 的 tag。
- 非空 → `git tag -a vNEW -F <临时文件路径>`。
- 无论成功失败，最后用 `fs.rmSync` 清理临时文件（放在 finally 或显式清理）。

**7. 推送**
- `git push origin master`
- 成功后 `git push origin vNEW`（推 tag 触发 GitHub `release.yml` 出包）。

**8. 失败处理**（贯穿全程，每一步包裹错误捕获）
- 前置检查阶段失败 → 直接退出，未改任何文件，提示简洁即可。
- bump / commit 阶段失败 → 提示当前已改/已提交到哪一步，给出 `git checkout -- <files>` 或 `git reset` 类回滚指引。
- 关键场景：**commit 已成功但 tag 或 push 失败** → 必须明确打印「已停在哪一步、当前仓库状态、如何手动收尾」。例如：
  - tag 已打但 `git push origin master` 失败 → 提示手动 `git push origin master` 后再 `git push origin vNEW`。
  - master 已 push 但 tag push 失败 → 提示手动 `git push origin vNEW`。
  - tag 已打、master+tag 均未 push → 提示手动 `git push origin master vNEW`。

**9. 输出**
- 全程关键步骤打印中文进度提示（如「✓ 前置检查通过」「✓ 已 bump 四处版本字段到 NEW」「✓ 已提交」「✓ 已打 tag vNEW」「✓ 已推送，release.yml 将自动出包」）。
- 正确处理 Windows：路径用 `node:path` 拼接、编辑器进程调用、读写不破坏 CRLF。
- 所有 `git` 调用建议封装一个 helper（`execSync` / `spawnSync`，`cwd` 固定为仓库根，即脚本目录的上一级）。
