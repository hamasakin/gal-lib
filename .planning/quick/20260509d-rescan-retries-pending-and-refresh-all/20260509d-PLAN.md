# Quick Task: rescan-retries-pending-and-refresh-all

## Goal

Two coordinated improvements so the user doesn't have to manually
re-pick metadata for every 「待复核」 game after the matching pipeline
gets stronger:

1. **Incremental scan auto-retries 待复核** — existing_paths in
   start_scan currently includes every games row, so re-running
   incremental skips a directory even when its metadata is stale/none.
   Filter to bound-only rows so the unbound ones get reprocessed.

2. **Force-refresh-all button** — bulk version of the per-row
   "重新匹配元数据" context-menu item. Useful after a cleaning/scoring
   pipeline change (like 20260509c) lands and the user wants to retry
   everything in one click.

## Non-goals

- No DB schema change; the metadata_source column already encodes the
  bound/none distinction we need.
- No new progress event type; reuse `scan-progress` for refresh_all.
- No partial-refresh granularity (e.g. "refresh only bound-with-low-conf").
  If users want that later, the `match_confidence` column is there, but
  shipping the simple force-all + auto-retry-none covers the user's ask.

## Tasks

### Task 1 — Incremental scan: existing_paths excludes unbound rows

**Files:**
- src-tauri/src/commands.rs (existing_paths SELECT)

**Action:**
Change the SELECT in `start_scan` from:
```sql
SELECT path FROM games
```
to:
```sql
SELECT path FROM games WHERE metadata_source IN ('bangumi','vndb','manual')
```

Side effect: incremental scan now re-ingests every directory whose
row has `metadata_source='none'` — re-discovering means rebuilding
the DiscoveredGame (fresh exe pick, fresh title clean) and running
the metadata pipeline again. Existing UPSERT semantics in
`ingest_one_dir` mean the row is updated in place (not duplicated).

**Verify:**
- Set up a fixture row manually with metadata_source='none' (or seed one
  via the example tool). Run incremental scan. Row's name/metadata fields
  refresh; bound rows untouched.
- `cargo build --lib` clean.

**Done:** When the user fixes the matcher and re-runs incremental, every
「待复核」 card is silently re-attempted; bound cards untouched.

### Task 2 — refresh_all_metadata Tauri command + Settings button

**Files:**
- src-tauri/src/commands.rs (new command)
- src-tauri/src/lib.rs (handler registration)
- src/lib/scan.ts (wrapper)
- src/routes/Settings.tsx (button + AlertDialog)

**Action:**
1. **commands.rs::refresh_all_metadata(app, state)** — list every game
   id+path+name+executable_path, emit a `scan-progress` Running event with
   `total=N, completed=0`, then for each game:
   - call `ingest::refresh_for_query(id, data_dir, path, name, exec)`
   - run the same UPDATE that `refresh_metadata` does
   - emit a Running event with `completed=i+1`
   At the end, emit a Completed event. Run inside `tokio::spawn` so the
   command returns immediately (mirrors `start_scan`).
2. **lib.rs** — register the command.
3. **scan.ts** — `export async function refreshAllMetadata(): Promise<void>`.
4. **Settings.tsx** — add a third button after 增量扫描:
   ```tsx
   <AlertDialog>
     <AlertDialogTrigger asChild>
       <SettingButton>强制刷新全部元数据</SettingButton>
     </AlertDialogTrigger>
     <AlertDialogContent>
       ... 此操作会重新搜索所有游戏的 Bangumi/VNDB 数据，含已绑定的。
       <AlertDialogAction onClick={() => void onRefreshAll()} />
     </AlertDialogContent>
   </AlertDialog>
   ```
   Update the Section lede to mention the new behavior.

**Verify:**
- `cargo build --lib` + `pnpm typecheck` clean.
- Wire-test: button click → confirm → toast "已启动元数据刷新" → progress
  bar advances → toast "刷新完成" once Completed.

**Done:** Single click bulk-refreshes; respects same rate limiters as
per-game refresh; bound games' rows update in place.

## Test plan (after both commits)

2. **Incremental retry:** 设置 → 增量扫描. Watch console — 待复核 cards
