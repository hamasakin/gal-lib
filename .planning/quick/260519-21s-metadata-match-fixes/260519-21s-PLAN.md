复现：右键条目『打开目录』（或 Detail『更多』→『打开本地目录』）→ 同一条目『重新匹配元数据』→ MetadataPicker 查询完成后会再弹出一个文件管理器窗口。

**第一步 —— 定位根因（读代码 + 推理，必要时插临时 console.log 验证）：**

后端 `open_in_explorer`（commands.rs:3373）已确认是无状态纯函数，根因不在后端，在前端 Radix 菜单 ↔ Dialog 的交互上。重点排查以下嫌疑链路：

- GameCard 右键菜单是 Radix `ContextMenu`，`打开目录` 与 `重新匹配元数据` 是相邻 `ContextMenuItem`，都用 `onClick`。`重新匹配元数据` 的 `onClick` 调 `onPickMetadata(game)` 打开 Dialog（MetadataPicker，modal）。
- 已知 Radix 交互陷阱：从菜单项打开一个 modal `Dialog` 时，菜单关闭 + Dialog 打开；当 Dialog 之后关闭，Radix `Dialog` 默认 `onCloseAutoFocus` 会把焦点交还触发元素（菜单 Trigger），此时菜单的「上一个 active/hover 项」可能被一个残留的合成 pointer/focus 事件重新激活 —— 如果上一个被激活过的项是 `打开目录`，它的 `onClick` 就会再次跑，于是又弹一次文件管理器。
- 也排查：`重新匹配元数据` 菜单项用 `onClick` 而非 Radix `onSelect`；`onClick` 不会被 Radix 的「选中后自动关菜单 + 焦点恢复」逻辑正确管理，可能导致菜单关闭时机/焦点恢复与 Dialog 打开竞争。
- Detail.tsx 路径同理：`DropdownMenuItem` 的 `打开本地目录` 和 `重新匹配元数据` 都用 `onClick`；`onClosePicker()`（Detail.tsx:798）在 picker 关闭后会跑 `refreshGame()` 等，但本身不调 `openGameDir` —— 确认它不是直接元凶，元凶是菜单焦点恢复重放 `打开目录` 的 onClick。

执行者需通过读代码 + （如有必要）临时打印 `onOpenDir` / `打开目录 onClick` 的调用栈与触发次数，**确认到底是哪条链路、哪个事件**让 `open_in_explorer` 被第二次 invoke。把确认到的根因一句话写进 SUMMARY。

**第二步 —— 根治修复（按确认到的根因选对应方案，不要全都套上）：**

最可能的正确修复（确认根因后采用）：
- 把 GameCard `ContextMenuItem` 和 Detail `DropdownMenuItem` 上的 `onClick` 改为 Radix 的 `onSelect`，让 Radix 统一管理「选中 → 关菜单 → 焦点恢复」生命周期，避免 `onClick` 与菜单关闭/Dialog 打开竞争导致的事件重放；
- 和/或：给 MetadataPicker 的 `Dialog`（或 DialogContent）加 `onCloseAutoFocus={(e) => e.preventDefault()}`，阻止 Dialog 关闭时把焦点强行甩回菜单 Trigger 从而重放上一个菜单项的激活。

不要用「加防抖 / 加标志位挡住第二次 openGameDir」这类掩盖症状的补丁 —— 要从事件链路上断掉重复触发。修复后两条入口（GameCard 右键、Detail『更多』）都要验证不再复发。

**第三步 —— 清理：** 移除定位阶段插入的所有临时 `console.log` / 调试代码。

需求：元数据匹配的置信度最低门槛降到 0，使最差候选也允许用户手动选中并应用。

**先定位**："门槛"在本项目里有两层语义，执行者读代码确认实际限制在哪：

1. `src-tauri/src/metadata/match_score.rs` 约 62 行 `let baseline: u8 = if is_prefix { 80 } else { 70 };` —— 这是非完全匹配场景的**最低保底分**。把它降到 0 意味着弱匹配候选会拿到接近 0 的真实低分，而不再被人为抬到 70/80。

2. MetadataPicker.tsx 的选中/应用逻辑：审查 `onApply()` 与候选 `<button>` 的 `onClick`（MetadataPicker.tsx 134-188、280-344 行）—— 确认**前端没有对 confidence 做下限过滤 / 禁用低分候选**。从已读代码看，前端候选 `<button>` 对所有候选都可点选，`onApply` 也不校验 confidence；`confidenceBadge` 仅控制 badge 颜色（<70 显示 destructive 红），不拦截选择。

**修复**：
- 把 match_score.rs 的 `baseline` 从 `if is_prefix { 80 } else { 70 }` 改为 `0`（即门槛降到 0）。保留其余评分逻辑（Levenshtein、prefix 判定等）—— 真实算得的分该是多少就是多少，只是不再有人为下限抬升。改完后该函数对最弱匹配会返回接近 0 的 confidence。
- 同步更新 match_score.rs 里因 baseline 改动而失效的**单元测试**（约 129、172-198 行的 `assert!(s >= 70 ...)` / `assert!(s >= 80 ...)` / `assert!(s >= 70 && s < 80 ...)` 等断言）：这些断言原本依赖 70/80 保底，baseline 改 0 后需相应放宽 —— 改为验证「prefix 命中仍 > 非 prefix」「完全匹配仍接近 100」这类相对关系，而非绝对 70/80 下限。把每条改动的测试断言改成与新语义一致且仍有意义的检查。
- 注意：types.rs:82-83 和 metadata.ts 的注释提到「≥80 auto-bind」。auto-bind（自动绑定）门槛是 ingest 阶段的独立逻辑，**本次不动 auto-bind 门槛**——只降匹配评分保底门槛。若 baseline 降 0 后注释里「≥80 auto-bind」的描述仍成立则不必改注释；如执行者发现 baseline 同时被 auto-bind 判定复用，需在 SUMMARY 说明并仅在确有耦合时谨慎处理，保持 auto-bind 行为不回归。

不要改前端（除非定位发现前端确有 confidence 下限过滤 —— 当前已读代码未发现，若发现再一并放开并写进 SUMMARY）。
