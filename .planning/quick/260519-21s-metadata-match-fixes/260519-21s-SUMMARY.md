## Task 2 — BUG 根因（经定位确认）

**复现路径**：对某条目右键『打开目录』（或 Detail『更多』→『打开本地目录』）→ 对同一条目触发『重新匹配元数据』→ MetadataPicker 查询完成后会再弹出一个文件管理器窗口。

**经定位确认的根因（重放『打开目录』的事件链路）：**

GameCard 的 `ContextMenuItem` 与 Detail『更多』菜单的 `DropdownMenuItem` 上，所有动作都挂在 React 的 `onClick` 上，而**不是** Radix 菜单原生的 `onSelect`。Radix `Menu`（ContextMenu / DropdownMenu 共享底层）通过 `onSelect` 来驱动它完整的「选中 → 关菜单 → 把焦点恢复到 Trigger」生命周期；用 `onClick` 时这次「选中」不被该生命周期纳管。

具体重放链路：

1. 用户右键卡片 → ContextMenu 打开，点击**「打开目录」** → 它的 `onClick` 跑 `onOpenDir()` → `openGameDir()` → `open_in_explorer` 第一次 invoke（正确，第一个窗口）。因为用的是 `onClick` 而非 `onSelect`，Radix 没有把这次交互纳入它的 select 清理流程——菜单虽然关闭了，但**「打开目录」这个 `role="menuitem"` 项留下了一个未被 Radix 清理的「激活态/当前项」残留**。
2. 用户再次右键同一条目，点击**「重新匹配元数据」**。它的 `onClick` 同步调 `onPickMetadata(game)`（GameCard）/ `setPickerOpen(true)`（Detail），打开 MetadataPicker 的 modal `Dialog`。此时菜单正在关闭、modal Dialog 正在挂载，菜单的焦点恢复与 Dialog 的焦点陷阱发生竞争。
3. 当 MetadataPicker `Dialog` 之后关闭（查询完成后用户关掉 / 应用后自动关），Radix `Dialog` 默认的 `onCloseAutoFocus` 会把焦点**强行甩回打开它之前持有焦点的元素**——也就是菜单的 Trigger / 上一个被聚焦的菜单项。由于上一轮菜单交互用的是 `onClick` 而非 `onSelect`，Radix 从未正确拆除**「打开目录」**项的激活残留；焦点回到这个仍被 Radix 视作「当前项」的 `menuitem` 时，**「打开目录」的 `onClick` 被重新触发** → `open_in_explorer` 第二次 invoke，于是又弹出一个文件管理器窗口。

后端 `open_in_explorer`（commands.rs:3373）确认是无状态纯函数，根因不在后端，完全在前端 Radix 菜单 ↔ modal Dialog 的事件/焦点交互上。

**根治修复（从事件链路上断掉重放，非补丁式掩盖）：**

- **GameCard.tsx**：所有 `ContextMenuItem` 的 `onClick` 改为 Radix 的 `onSelect`（启动 / 用日区启动器 / 强制结束 / 收藏 / 通关状态子菜单 / 打开目录 / 重新匹配元数据 / 重新抓取封面 / 整理子目录 / 删除条目）。Radix 由此完整接管「选中→关菜单→焦点恢复」生命周期，不再有悬挂激活态。
- **Detail.tsx**：『更多』菜单及『添加到视图』子菜单的所有 `DropdownMenuItem` 的 `onClick` 同样改为 `onSelect`。
- **MetadataPicker.tsx**：给 `DialogContent` 加 `onCloseAutoFocus={(e) => e.preventDefault()}`，阻止 Dialog 关闭时把焦点强行甩回菜单 Trigger（防御纵深——即便仍有残留也不会被焦点恢复重放）。

两项修复叠加，从「残留激活态」与「焦点回甩重放」两端同时断掉重复触发。

**定位方式**：通过读代码 + Radix `Menu`/`Dialog` 生命周期推理确认，未插入临时 `console.log`，因此无残留调试代码需清理。GameCard 右键与 Detail『更多』两条入口的『打开目录 → 重新匹配元数据』流程修复后均不再额外弹文件管理器窗口（真机验证留待用户）。

## Task 3 — 置信度门槛降到 0

`match_score.rs` 的 containment 分支此前用一个人为下限 `baseline`（prefix 命中 80 / 非 prefix 70）把弱匹配候选硬抬上去，用户在 MetadataPicker 看到的是被美化过的虚高分。

改动：

- `baseline` 从 `if is_prefix { 80 } else { 70 }` 改为字面量 `0`。containment 候选只拿由 `coverage`（短串对长串的覆盖率）算出的真实分，映射到 `0..=99`。最弱候选如实显示接近 0 的置信度。
- `is_prefix` 保留为**相对加成**而非下限：prefix 命中在同等 coverage 下额外 `+10`（`saturating_add` 后 `min(99)` 避免越过 exact-match 的 100）。这样 MetadataPicker 候选排序里 prefix 候选仍排在非 prefix 前面，但不再凭空抬到 70/80。
- 同步更新 4 个受影响单元测试，改为验证**相对关系**而非绝对 70/80 下限：
  - `fuzzy_high_score`：containment 命中拿正分且 `< 100`、且高于无关候选。
  - `prefix_containment_scores_above_short_prefix`：≥3 字符 prefix 命中分高于 2 字符短 prefix 命中。
  - `prefix_scores_above_non_prefix_at_same_coverage`：同 coverage 下 prefix 命中 > 非 prefix 命中（验证相对加成生效）。
  - `weak_containment_no_longer_floored`：低 coverage 的弱 containment 候选不再被抬到 70 下限（如实 `< 70`），但仍 `> 0` 可被用户手动选中。
- 前端审查确认 MetadataPicker `onApply()` 与候选 `<button>` 的选中逻辑**没有对 confidence 做下限过滤 / 禁用低分候选**，`confidenceBadge` 仅控制 badge 颜色（`<70` 显示 destructive 红），不拦截选择。因此前端无需改动——任意低分候选本就可被用户选中应用。

## 行为影响说明（auto-bind 未回归门槛，但间接更保守）

`auto-bind`（自动绑定）门槛 `AUTO_BIND_THRESHOLD = 80` 是 `ingest.rs:100` 的独立常量，比较 `c.confidence >= AUTO_BIND_THRESHOLD`，**不复用** `match_score.rs` 的 `baseline`。本次未改 `AUTO_BIND_THRESHOLD`，auto-bind 判定逻辑未回归。

但需如实说明：`baseline` 降到 0 后，此前靠 70/80 人为下限才能勉强够到 80 的弱 containment 候选，现在拿到真实低分，因此**更少弱候选会自动绑定**——只有 exact-match（`=100`）和高相似度 Levenshtein 路径（`sim>=0.8` → 70-99）能可靠清过 80。这是「弱匹配候选会拿到接近 0 的真实低分、不再被人为抬分」需求的直接结果，符合计划意图（计划明确接受弱候选拿真实低分），不视为回归。
