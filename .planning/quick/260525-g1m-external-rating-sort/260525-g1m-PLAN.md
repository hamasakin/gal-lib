## Bangumi `GET /v0/subjects/{id}` 的 rating 字段形状
```json
{
  "rating": {
    "rank": 123,
    "total": 1234,
    "count": { "1": 0, "2": 0, ..., "10": 99 },
    "score": 7.8
  }
}
```
- `rating.score` 是 0..=10 的 float（1 位小数）
- `rating.total` 是参与打分的人数（i64）
- 已在 `SubjectDetail` 反序列化结构（`bangumi.rs:331`）但**当前未捕获 rating 字段** — 执行器需新增子结构 + `#[serde(default)] rating: Option<RatingObj>` 并在 `fetch_detail` 中读出 score / total

## VNDB `/kana/vn` 的 rating 字段形状
- API doc: rating 是 `0..=100` 的 float（1 位小数；2 位小数会被四舍五入），`votecount` 是 i64
- VNDB 在 `fields(...)` 字符串里**必须显式列出**才返回 — 当前 `vndb.rs:126-128` 的 `fields` 没有 `rating,votecount`，需要追加
- **归一化**：VNDB 0..=100 / 10 → 0..=10 存入 `external_rating`（与 Bangumi 同口径，前端不分源直接 toFixed(1)）

**重要 — 不要混淆**：`vndb.rs:127` 的 `tags{name,rating,spoiler,category}` 里的 `rating` 是**标签权重**（0..=3），与本次的「VN 整体评分」不是同一字段。`DetailHit` 顶层加 `rating: Option<f64>` 和 `votecount: Option<i64>`，与 `TagEntry.rating` 各自独立。
