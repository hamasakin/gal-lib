<decisions>
## Implementation Decisions

### 子任务 1：取消扫描修复

- **根因**：`commands.rs::start_scan` 的 ingest 循环（line 396-442）整个循环里没有 `ctx.cancel.load()` 检查；`scan/mod.rs::run_scan` 只在 Pass 1 (discover) 检查。`refresh_all_metadata` 的循环同理（commands.rs ~605-722）。
- **修法**：在 ingest 循环每次迭代顶部、`refresh_all_metadata` 循环每次迭代顶部，加一次 `if ctx.cancel.load(Ordering::Relaxed) { emit Cancelled; return; }`。
- **前提**：`refresh_all_metadata` 当前没用 `ScanContext`（看 commands.rs 验证）—— 如果没有，需要让它共享同一个 `ScanState` mutex 里的 ctx，cancel_scan 命令同时能取消两条路径。**采纳**：`refresh_all_metadata` 也走 `ScanState` 注入 ctx。
- **取消事件**：当前 `cancel_scan` 命令本身只设 flag 不 emit；真正的 `ScanStatus::Cancelled` 事件由 ingest 循环的 cancel-check 分支负责 emit。

### 子任务 2：跨 game ingest 并发 + 跨 game query dedup

**已经是这样的（现状，不动）：**
- 同 game 内 Bangumi + VNDB 是 `tokio::join!` **并发**查（`ingest.rs::pick_best_across_sources` line 68-83）—— v1.0 设计，文件头注释明确
- aggressive_candidates fan-out（commit 4773460）在同 game 内 sequential 跑 N 个 candidates × `pick_best_across_sources` —— **保留 sequential**（用户决策："同游戏不用并发"）
- Bangumi/VNDB 限速器是 governor token-bucket process-wide singleton（`metadata/limiter.rs`），**已经线程安全**

**新增（要做的）：**
- **跨 game ingest 并发**：commands.rs::start_scan 的 ingest 循环改为 `tokio::task::JoinSet` 或 `futures::stream::iter(...).buffer_unordered(N)`；并发数 `INGEST_CONCURRENCY = 4`（const，不引入 config 字段）
- **跨 game query dedup cache**：`Arc<Mutex<HashMap<String, Arc<tokio::sync::OnceCell<Option<Candidate>>>>>>`，scope = 单次 start_scan 调用（不跨 scan 持久化，每次 start_scan spawn 新 cache）
  - Key：cleaned query string（同 `pick_best_across_sources` 的入参）
  - 命中：复用前次 search 结果（`Option<Candidate>` 是 already-merged + filtered 的最优 candidate）
  - 防 thundering-herd：用 `OnceCell::get_or_init` 让多个并发 task 命中同一 query 时只发一次请求
  - LRU/上限：单次 scan 内 entries 数 ≤ 116 game × ~2 candidates = ~230，不需要上限
- **`pick_best_across_sources` 不动**；新增 `pick_best_across_sources_cached(query, &cache)` wrapper，`process_game` / `refresh_for_query` 接受可选 `Option<&QueryCache>` 参数（None = 不走 cache，保持 add_game / refresh_metadata / bind_metadata 单条路径不变）
- 或者更简单：把 cache 作为 ingest 模块的 thread-local-style 参数，仅 start_scan 路径传入；其他单条 entry-points 不传

**并发安全注意点**：
- `scan-progress.completed` counter：当前用 `enumerate` 的 `i+1`，并发后顺序乱 → 改为 `Arc<AtomicUsize>`，每 task 完成后 `fetch_add(1, Relaxed)`，emit 用当前值
- `current_dir` 字段：并发后乱序，UI 上"正在抓 X"会跳跃 —— **接受**，这是并发的代价；用户已有 meta-fetch-progress 的 per-game 高亮，不依赖 current_dir
- 占位行 INSERT (`insert_placeholder_dir`)：已经幂等（`ON CONFLICT(path) DO UPDATE`），并发安全
- Cancel flag 在每个并发 task 顶部检查，确保 cancel 时所有 in-flight task 优雅退出（每 task 不再发新请求；正在 await 的 token-bucket 不可中断，最差会再发 N 个 request 才退出，可接受）

### 子任务 3：GameGrid 重启虚拟化

- **现状**：`@tanstack/react-virtual ^3.13.24` 已在 deps；`GameGrid.tsx` v1.0 用过现已移除（注释说 "Re-add virtualization if 1000+ games"）
- **重启策略**：用 `useVirtualizer` row-mode（按行虚拟化），manual lane indexing 计算 `columnCount`（因为 grid 是 `auto-fill, minmax(--card-w, 1fr)`，需要 ResizeObserver 测量 container width / card width 推 cols）
- 02f Decisions 已经描述过这个 pattern：「row-mode with manual lane indexing (count = ceil(games/cols)); columnCount derived from ResizeObserver」—— 直接复刻
- estimateSize：固定 `--card-w * 4/3 + 56px` (cover 高 + meta 高 + gap)，避开测量
- overscan：5 行（默认 1 太少，扫动会闪）
- 不影响：DensityToggle 切换 `--card-w` 时，ResizeObserver 触发 columnCount 重算 → useEffect 重新跑 → 虚拟化器重 layout

### Claude's Discretion
- INGEST_CONCURRENCY 具体值（4 是 default 建议，可调）
- query cache 注入方式（参数 vs lazy_static vs OnceCell）—— 推荐参数注入，单次 scan 内 scope，自动 drop
- atomic counter 的 emit 节流（如果 4 task 并发完成可能在 1ms 内 emit 4 次 scan-progress；可加节流也可不加）
- react-virtual 估算 row height 的精确算法（cover aspect 3/4 + meta + padding）
- ScrollContainer ref 是放 GameGrid 内还是从 Library.tsx 外层传（Library.tsx 我刚改了 `<div flex flex-1 overflow-auto>` 包整个 grid 区，这个 div 就是 scrollContainer——GameGrid 内部不再需要自己的 overflow-auto）
