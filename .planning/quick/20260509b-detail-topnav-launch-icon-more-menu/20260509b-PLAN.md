# Quick Task: detail-topnav-launch-icon-more-menu

## Goal

Three coordinated detail-page polish items:

1. **Top nav bar** above the hero, containing the back button (with
   Esc keyboard hint), a breadcrumb, and prev/next siblings navigation.
   Removes the duplicate back-button + breadcrumb-ish line currently
   embedded in the hero info column.
2. **Launch-button play-icon centering.** The right-pointing Play
   triangle has its visual centroid offset to the left of its bounding
   box; in the collapsed 44px state the icon appears too far left.
3. **更多 dropdown.** The inert MoreHorizontal button next to the
   launch button becomes a real shadcn DropdownMenu with 打开本地目录
   and 复制路径 entries.

## Why

- Top bar consolidates navigation surfaces — Esc, breadcrumb,
  list-position arrows — that are otherwise scattered or absent. The
  brand mini-nav inside hero ("← 图书馆 · {brand} · {year}") was a
  partial breadcrumb that didn't actually link the brand or game name.
- Visual centering of the play triangle is a well-known icon affordance
  problem. A small `translate-x` offset balances perceived weight.
- The MoreHorizontal button has lived empty since v1.1 ship and the
  detail page lacks a quick way to jump to the actual game folder on
  disk — the most-asked-for path for galgame users (manual save fixes,
  patch installs, cracking issues).

## Out of scope

- Sidebar 路径 section's 复制路径 / 封面源 buttons stay where they are
  — adding the 复制路径 entry to the dropdown is a duplicate by design
  (the dropdown is for the hot path; the sidebar is for path inspection).
- No "下一首" / "上一首" keyboard arrows yet — only mouse-clickable
  arrows in the top bar. Keyboard arrow shortcuts can be a follow-up.
- No animation polish on the dropdown itself — use shadcn defaults.
- Bangumi / VNDB search-source buttons stay in the sidebar; they're not
  duplicated in the dropdown.
- Focus the open-in-explorer feature on Windows (project is Windows-only
  per CLAUDE.md). `explorer.exe {path}` is the only path tested.

### Task 1 — `open_in_explorer` Tauri command

**Files:**
- `src-tauri/src/commands.rs` (append new command)
- `src-tauri/src/lib.rs` (register in invoke_handler)
- `src/lib/games.ts` (frontend wrapper)

**Action:**

Backend command:

```rust
/// Open the given filesystem path in the OS file manager (Windows
/// Explorer). Used by the detail-page 更多 menu's 打开本地目录 entry.
///
/// Path validation: rejects non-existent paths so a stale `games.path`
/// (deleted directory) surfaces a clean error instead of explorer's
/// generic "location not available" dialog. Beyond that, the path comes
/// from `games.path` which the user provided themselves at scan time —
/// no further sanitisation is necessary, and `Command::arg()` passes the
/// path as a separate argv entry (no shell interpretation, no command
/// injection).
#[tauri::command]
pub fn open_in_explorer(path: String) -> Result<(), String> {
    use std::path::Path;
    use std::process::Command;
    if !Path::new(&path).exists() {
        return Err(format!("路径不存在：{}", path));
    }
    Command::new("explorer")
        .arg(&path)
        .spawn()
        .map(|_| ())
        .map_err(|e| format!("无法打开 Explorer：{}", e))
}
```

Register in `lib.rs` `invoke_handler!`:

```rust
commands::open_in_explorer,
```

Frontend wrapper in `src/lib/games.ts`:

```ts
/**
 * Open a filesystem path in the OS file manager (Windows Explorer).
 * Errors propagate as toast-friendly strings ("路径不存在...").
 */
export async function openGameDir(path: string): Promise<void> {
  await invoke("open_in_explorer", { path });
}
```

### Task 2 — Detail page top nav bar

**Files:** `src/routes/Detail.tsx`

**Action:**

Add a new section ABOVE the existing hero `<section>`:

```jsx
<TopBar
  game={game}
  position={position}
  total={total}
  onBack={() => navigate("/")}
  onPrev={prevId ? () => navigate(`/games/${prevId}`) : null}
  onNext={nextId ? () => navigate(`/games/${nextId}`) : null}
```

`TopBar` is a local component with this layout:

```
┌─────────────────────────────────────────────────────────┐
│ ← 返回图书馆 [Esc]   图书馆 / {brand} / {title}     ← N/M → │
└─────────────────────────────────────────────────────────┘
```

- Height: 52px.
- Border-bottom: 1px line, bg-bg-1 / bg-bg-0 backdrop.
- Sticky to top: `sticky top-0 z-30` so it stays visible while the
  hero scrolls.
- Back button: ghost-styled pill, Esc kbd badge using existing
  `<kbd>`-style class (font-mono text-[10px] border bg-bg-2).
- Breadcrumb: 3 segments with `/` separators.
  - 图书馆 → click navigates to `/`.
  - {brand} → click sets filter.brand and navigates `/`. Hidden when
    `game.brand` is null.
  - {title} → current page (no link, ink-1 color).
  - Truncates with ellipsis when narrow.
- Prev/next arrows: 28×28 ghost icon buttons + `N/M` font-mono
  count between them. Disabled state when no sibling.

Keyboard handler — useEffect listening for Esc:

```ts
useEffect(() => {
  const onKey = (e: KeyboardEvent) => {
    if (e.key === "Escape") {
      e.preventDefault();
      navigate(-1);
    }
  };
  window.addEventListener("keydown", onKey);
  return () => window.removeEventListener("keydown", onKey);
}, [navigate]);
```

## Risks / pitfalls

- **Sticky top bar + scrollable container.** Detail.tsx's outer wrapper
  is `<div className="relative h-full w-full overflow-auto">` — the
  scroll container. `sticky top-0` works *inside* this wrapper because
  it's the scroll context. Confirmed by reading the existing layout.
- **Prev/next when navigating from a deep link.** If user opens
  `/games/42` directly (no Library route mount yet), `useLibraryStore.
  games` is empty and arrows disable. Not a regression — same UX as
  before because there's no list to navigate. Acceptable.
- **Esc handler conflict.** Existing dialogs (AlertDialog, MetadataPicker,
  Lightbox) all use Radix Dialog primitives, which trap focus and
  consume Esc internally. Our window-level Esc listener fires only when
  no Radix dialog is open (Radix calls stopPropagation on Esc). Confirmed
  by reading shadcn dialog.tsx earlier in the session. Tested by spec only.
- **`Command::new("explorer")`.** On Windows, the canonical way to
  open a folder is `explorer.exe <path>`. PATH normally includes
  `C:\\Windows`. If a user has corrupted PATH, the spawn fails and we
  surface "无法打开 Explorer". Accepted edge case.
- **Backend 44 vs 43 command count.** Bumping to 44. Update STATE.md
  decision log via SUMMARY (not via a rewrite of phase-05 entries).
