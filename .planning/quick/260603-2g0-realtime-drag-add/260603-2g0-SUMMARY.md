# Quick 260603-2g0 — 拖曳目录添加游戏实时刷新（SUMMARY）

## 做了什么

让资源管理器拖拽目录添加游戏接上项目现成的实时刷新管线，占位卡瞬间出现 +
元数据逐个回填 + 落点即时 loading 反馈。

### 改动

1. **`src-tauri/src/commands.rs`** — 新增 `add_games(paths)` 批量命令 +
   `AddGamesResult { added, failed }`。两阶段：
   - Phase 1：遍历所有路径，校验（`is_dir` + `ensure_safe_game_dir`）后
     `insert_placeholder_dir`，每条 `emit("games-changed")` → 所有占位卡秒出。
   - Phase 2：逐个 `enrich_metadata_for_dir`，前后 `emit("meta-fetch-progress"
     started/finished)` + `emit("games-changed")` → 卡片逐个回填 + 脉冲高亮。
   - enrich 失败仅 `eprintln` 记录、不回退（占位行保留），不减 `added`。
2. **`src-tauri/src/lib.rs`** — `invoke_handler` 注册 `commands::add_games`。
3. **`src/lib/scan.ts`** — 新增 `addGames(paths)` invoke 包装 + `AddGamesResult` 类型。
4. **`src/App.tsx`** — drop 循环从 `for (p) await addGame(p)` + 整批末刷新，改为
   单次 `await addGames(paths)`；落点立即 `toast.loading(dragdrop.adding)`，结束转
   success/partial。保留结束后的兜底 `searchGames` 刷新（覆盖落点时不在 Library
   路由的场景）。移除 `addGame` import。
5. **i18n** — 三语（zh/en/ja）新增 `dragdrop.adding` loading 文案。

## 为什么这样改

`Library.tsx` 已订阅 `games-changed` 节流刷新网格，`main.tsx` 已订阅
`meta-fetch-progress` 做 per-card loading 脉冲——全量扫描 `start_scan` 就靠这套做
渐进展示。拖拽之前完全没接上（`add_game` 不接 `AppHandle`、不发事件）。复用现成
管线，前端几乎零额外接线。

## 验证

- `cargo check`（src-tauri）：0 error。
- `pnpm typecheck`（tsc --noEmit）：exit 0。
- `cargo build` 仅在最后替换 `gal-lib.exe` 时「拒绝访问」——因 dev app 正在运行锁住
  exe，非代码问题（lib 已编译通过）。

## 待人工真机验证（子代理/CLI 跑不了 GUI）

- 多选多个目录拖入：占位卡是否**瞬间全部**出现（带「获取中」），随后逐个回填封面/标题。
- 落点是否立即出现「正在添加目录…」loading toast，结束后转「已添加 N 个」。
- 含无效路径（非目录/系统目录）时 partial 文案 `已添加 X 个,Y 个失败` 是否正确。
